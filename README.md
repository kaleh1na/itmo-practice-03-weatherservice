# Курс "AI-инструменты в жизни инженера: от идеи до продакшена" | ИТМО

## 📋 Executive Summary

Интенсивный 8-недельный практический курс, обучающий инженеров-разработчиков системному применению AI-инструментов на всех этапах жизненного цикла разработки ПО. 

**Основа курса:** Engineer Journey Map (EJM) — каркас, который связывает AI-инструменты с реальными этапами SDLC: требования → дизайн → реализация → тестирование → CI/CD → эксплуатация.

**Сквозной проект:** "PythonNotify" — Telegram-бот для уведомлений о погоде, развивающийся от идеи до production.

**Финальная цель:** Формирование мышления **"AI-augmented engineer"** — инженера, который не просто пишет код, а проектирует и автоматизирует процессы разработки, выступая в роли архитектора и технического руководителя для своих AI-ассистентов.

## 🎯 Философия курса

> **Фундаментальный сдвиг роли инженера: от исполнителя к архитектору**

### Новая парадигма инженера:

**Вы больше не "кодер", вы — архитектор и тимлид для AI:**

1. **Проектируете задачи** через промпт-инжиниринг
2. **Управляете контекстом** через MCP и структуру знаний
3. **Проводите строгое код-ревью** сгенерированного кода
4. **Автоматизируете процессы**, а не только код

### Ключевые принципы:

- **EJM как каркас:** Engineer Journey Map — не один урок, а сквозная структура всего курса
- **Проектно-ориентированное обучение:** Один сквозной проект через все 8 недель
- **Итеративное углубление:** От простых запросов к профессиональным техникам на тех же задачах
- **От локального к глобальному контексту:** От `@file` до полноценного MCP
- **Constitution-driven development:** у проекта есть «конституция» (`.specify/memory/constitution.md`) — неизменяемые принципы (язык, TDD, Library-First, CLI, integration testing)
- **Проверяемость и воспроизводимость:** каждый результат фиксируется артефактами (Markdown/диаграммы/тесты), а код живёт в репозитории с понятными правилами (`AGENTS.md`, `specs/`)

## 🧰 Инструменты и «рабочее место инженера»

- **Universal LLMs**: ChatGPT / Claude / Gemini / DeepSeek (в зависимости от доступности)
- **Провайдеры моделей**: OpenRouter/Vsellm (единая точка доступа к разным LLM)
- **Coding agents**: Cursor / VSCode + Roo Code (agentic-разработка в IDE)
- **Оркестрация и автоматизация**: N8N (локально или в облаке) для workflow и интеграций
- **Инженерный контур**: GitHub (Issues/PR), GitHub Actions (CI/CD), PlantUML/Miro/Mermaid (диаграммы), pytest/coverage (тестирование)

### Неделя 5: Расширение контекста и рефакторинг (Spec-driven development)

**Концепция:** Усложняем проект «по-взрослому»: расширяем контекст, добавляем API, запускаем локальный сервис и делаем рефакторинг без потери функциональности.

| Что изучаем | Результат |
|------------|-----------|
| Расширение контекста проекта | Добавляем карты/схемы/доп.артефакты для точности агента |
| Создание API и запуск локального сервиса | Первый «живой» запуск и ручная проверка в терминале |
| Рефакторинг без регрессий | Улучшаем структуру (в т.ч. под Library-First/CLI) |
| Новые режимы работы (планирование/имплементация/и т.п.) | Упорядочиваем процесс «контекст → задача → ревью → уточнение» |

**Практика:** Practice 05 - Spec-driven Development & Refactoring (в разработке)

---

### Неделя 6: Ассистент для код-ревью (diff → LLM → review в GitHub)

**Концепция:** Автоматизируем ревью PR: подключаем триггер, забираем diff, просим LLM сделать review по правилам проекта и публикуем результат в GitHub.

| Что изучаем | Результат |
|------------|-----------|
| Настройка триггера (PR/обновление PR) | Авто-запуск ревью по событию |
| Получение diff | Единый вход для анализа изменений |
| Подключение LLM для ревью | Структурированный review (риски, стиль, тесты, edge cases) |
| Публикация ревью в GitHub | Комментарий/Review с actionable items |
| Проверка ассистента на новой фиче | Делаем ещё одно изменение и валидируем workflow |

**Практика:** Practice 06 - AI Code Review Assistant (в разработке)

---

### Неделя 7: Пишем тесты с помощью AI (до ~90% покрытия unit-тестами)

**Концепция:** Ускоряем тест-дизайн и покрытие: от happy path к edge cases, затем закрываем пробелы и стабилизируем базовую бизнес-логику.

| Что изучаем | Результат |
|------------|-----------|
| Happy path | Минимальный набор «работает как задумано» |
| Edge cases | Поиск граничных сценариев и ошибок |
| Дописываем тесты под найденные кейсы | Стабильный набор unit-тестов |
| Добиваемся ~90% покрытия unit-тестами | Осознанное покрытие ключевой логики |

**Практика:** Practice 07 - AI-powered Testing (в разработке)

---

### Неделя 8: Автоматизируем поддержку — от баг-репортов к тикетам (+ метрики пользы)

**Концепция:** Принимаем неструктурированные баг-репорты, анализируем их LLM, создаём Issue в GitHub и формируем «новые требования» для следующего цикла разработки.

| Что изучаем | Результат |
|------------|-----------|
| Создаём workflow поддержки | Пайплайн обработки входящих сообщений |
| LLM для анализа баг-репорта | Структурирование: шаги, ожидание/факт, окружение, приоритет |
| Создание Issue в GitHub | Тикет с готовым описанием и чек-листом |
| Новые требования → новый цикл | Переход от поддержки к планированию следующей итерации |
| Бизнес-метрики для аргументации пользы | Показатели, объясняющие эффект автоматизаций |

**Практика:** Practice 08 - Support Automation & Triage (в разработке)

**Финальный проект:** Workflow, который принимает жалобу/баг-репорт, анализирует с помощью LLM и создаёт готовый Issue/тикет (с приоритетом, шагами воспроизведения и предложением решения).

## 🧪 Формат практик и семинаров (как на реальном проекте)

- **Семинар (практики-обсуждения)**: разбираем артефакты и требования (User Stories, DoR/DoD, AC, test plan), фиксируем улучшения.
- **Практика (hands-on)**: реализуем кусок функциональности сквозного проекта в коде, через цикл «контекст → задача → ревью → уточнение».
- **Сопровождение**: работаем с правилами проекта (Constitution/AGENTS/MCP), чтобы агент не «галлюцинировал», а изменения были воспроизводимыми.

## 📝 Домашние задания (лабораторные работы)

- **После лекции/семинара**: подготовить конспект/заметки и улучшить артефакты (DoR/DoD/AC/test plan) по итогам обратной связи.
- **Фича-итерации**:
  - Реализовать следующую User Story из своего бэклога (например, подписка/отписка) и покрыть её тестами.
  - Реализовать «третью» User Story с дополнительными правилами (валидации/ограничения) и оформить это через Spec-driven подход (спецификация → тесты → код).
- **Тестирование**: добавить API-тесты или E2E-тесты на ключевые пользовательские сценарии.
- **CI/CD**: добавить в свой pipeline матрицу сборок и базовые проверки качества.
- **Автоматизация**: улучшить свой workflow (N8N/Actions) — например, отправлять уведомление в чат/почту с заголовком тикета и ссылкой на него.

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel, field_validator

# Pydantic модель с валидацией
class SubscriptionRequest(BaseModel):
    city: str
    
    @field_validator('city')
    @classmethod
    def city_must_not_be_empty(cls, v: str) -> str:
        if not v or not v.strip():
            raise ValueError('City cannot be empty')
        return v.strip()
