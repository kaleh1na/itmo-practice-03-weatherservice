# Практика 02: Профессиональный промптинг с R.C.T.F.

**Связь с лекцией:** R.C.T.F. Framework (Role, Context, Task, Format)

## Цель практики

Освоить фреймворк R.C.T.F. для создания структурированных промптов профессионального уровня и улучшить артефакты из Практики 1, превратив хаотичное общение с AI в контролируемый инженерный процесс.

## Итоговые артефакты (что сдаём)

Артефакты оформляются в `practices/practice_02/task.py` и попадают в итоговый отчёт `artifacts/report_p2.md`.

Студенты создадут улучшенные версии (v2.0) артефактов из Практики 1 для проекта "WeatherService":

- Детализированная архитектурная схема в Mermaid (с Redis для кэша, PostgreSQL для БД)
- Формализованные требования в формате Gherkin (Given/When/Then)
- Улучшенные DoR и DoD (структурированные по категориям)
- Структурированный план тестирования по пирамиде тестирования
- Улучшенный Functional Delivery (детализированные Jira-тикеты)
- Полный журнал R.C.T.F. промптов с анализом результатов

### 3. Задание 2 — Формализация требований в Gherkin

### Цель блока

Превратить User Stories из Практики 1 в формальные требования в формате Gherkin (Given/When/Then).

### Что говорит преподаватель

- «Формализуйте. Сделайте требования пригодными для работы команды разработки и QA»
- «Gherkin — это язык описания поведения системы в формате Given/When/Then. Это понятно и разработчикам, и тестировщикам»
- «Проверяйте через негативные сценарии: что если город не найден? Что если API недоступен?»

### Что показывать на экране

- Пример Acceptance Criteria в Gherkin синтаксисе
- Структура: Feature → Scenario → Given/When/Then
- Требования к качеству (позитивные + негативные сценарии)

### Инструкции студентам

1. Выберите 2-3 User Stories из Практики 1
2. Составьте R.C.T.F. промпт для формализации в Gherkin
3. Расписать Acceptance Criteria в формате Given/When/Then
4. Указать позитивные и негативные сценарии
5. Зафиксировать финальный промпт в `RCTF_Log`

### Техники вовлечения

- «Критика через роль»: представьте, что вы Junior QA — задайте 3 вопроса к требованиям соседа
- Голосование за самый чёткий AC (Acceptance Criteria)

### Ожидаемый артефакт

- 2-3 User Stories, переписанные в формат Gherkin (.feature файлов)
- Каждый сценарий должен содержать позитивный и негативный случай
- R.C.T.F. промпт для генерации

### Пример промпта

```
Role: Ты опытный Business Analyst, специализирующийся на BDD (Behavior Driven Development).
Context: Для проекта WeatherService (REST API) нужно формализовать требования
         для команды разработки и QA.
Task: Преобразуй User Story "Как пользователь API, я хочу подписаться на погоду в городе через POST /subscribe"
      в формат Gherkin. Создай:
      - 1 позитивный сценарий (успешная подписка через API)
      - 2 негативных сценария (город не найден, email уже подписан)
Format: Gherkin синтаксис (.feature файл) с Feature, Scenario, Given/When/Then.
```

### 5. Задание 4 — Структурирование плана тестирования

### Цель блока

Создать профессиональный тест-план, структурированный по пирамиде тестирования (Unit, Integration, E2E).

### Что говорит преподаватель

- «Ваша цель — собрать документ, который QA реально сможет использовать»
- «Разделим тесты на три уровня: Unit (тестируем функции), Integration (связи между модулями), E2E (весь флоу)»
- «Добавляем предусловия, шаги, ожидаемый результат — как в настоящем тест-кейсе»

### Что показывать на экране

- Пример таблицы тестов (Test ID, Level, Preconditions, Steps, Expected Result)
- Пирамида тестирования (Unit → Integration → E2E)
- Критерии качества тест-плана

### Инструкции студентам

1. Возьмите текущую таблицу/план v1.0 из Практики 1
2. Составьте R.C.T.F. промпт для пересборки тест-плана
3. Структурируйте тесты по уровням (Unit, Integration, E2E)
4. Поддерживайте формат Markdown таблицы
5. Зафиксируйте промпт в `RCTF_Log`

### Техники вовлечения

- Быстрый "peer check": команды обмениваются 1 тест-кейсом — проверьте, все ли поля заполнены
- Вопрос: «Какой тест самый критичный для вашей системы?»

### Ожидаемый артефакт

- Структурированный тест-план в формате Markdown таблицы
- Минимум 6 тест-кейсов: 2 Unit, 2 Integration, 2 E2E
- Каждый тест-кейс содержит: ID, Level, Preconditions, Steps, Expected Result

### Пример промпта

```
Role: Ты Senior QA Engineer с опытом в тестировании REST API.
Context: Проект WeatherService (REST API) требует полного покрытия тестами.
Task: Создай тест-план, структурированный по пирамиде тестирования:
      - 2 Unit теста (тестирование функций валидации города, кэширования)
      - 2 Integration теста (интеграция с Weather API, работа с PostgreSQL)
      - 2 E2E теста (полный флоу: POST /subscribe → GET /weather/{city})
Format: Markdown таблица с колонками: Test ID | Level | Preconditions | Steps | Expected Result
```
