- `MERMAID_V2` — улучшенная архитектурная схема
- `GHERKIN_SCENARIOS` — формализованные требования в Gherkin
- `DOR_V2` — улучшенный Definition of Ready
- `DOD_V2` — улучшенный Definition of Done
- `TEST_PLAN_V2` — структурированный план тестирования
- `FUNCTIONAL_DELIVERY_V2` — улучшенные Jira-тикеты
- `HOMEWORK_EVENT_STORMING_V2` — опционально: улучшенный Event Storming
- `HOMEWORK_ROADMAP_V2` — опционально: улучшенный Roadmap
- `REFLECTION` — рефлексия

После завершения запустите:

```bash
python3 task.py
```

Сформируется отчёт: `artifacts/report_p2.md`

## Что такое R.C.T.F.?

**F**ormat — В каком виде выдать ответ? (Markdown, Gherkin...)

## Задание 1: Рефакторинг архитектуры

**Цель:** улучшить схему до продакшен-уровня

**Используйте R.C.T.F.:**

- **Role:** "Ты Senior Backend Architect"
- **Context:** "REST API с кэшированием"
- **Task:** "Добавь Redis для кэша, покажи взаимодействие с PostgreSQL"
- **Format:** "Mermaid диаграмма компонентов"

**Результат:** Детализированная Mermaid диаграмма v2.0

**Инструмент:** <https://mermaid.live/>

**Пример R.C.T.F. промпта:**

```
Role: Ты Senior Backend Architect 
с опытом проектирования REST API.
Context: Мы разрабатываем WeatherService — 
REST API для уведомлений о погоде.
Требуется кэширование данных о погоде 
для минимизации вызовов к внешнему API.
База данных — PostgreSQL для подписок 
пользователей.
Task: Улучши существующую архитектурную схему:
- Добавь Redis для кэширования данных 
о погоде (TTL 10 минут)
- Укажи, как PostgreSQL хранит подписки 
(таблицы: users, subscriptions)
- Покажи взаимодействие: Client → FastAPI → 
Redis/Weather API → PostgreSQL
- Укажи протоколы взаимодействия (HTTP REST)
Format: Mermaid диаграмма компонентов 
(graph TB или graph LR). Используй нотацию 
для компонентов и связей.
```

## Задание 2: Gherkin

**Цель:** превратить User Stories в формальные требования

**Формат:**

```gherkin
Feature: Weather Subscription API
  Scenario: Successful subscription
    Given клиент имеет валидный API endpoint
    When клиент отправляет POST /subscribe
    Then API возвращает 200 OK
```

**Результат:** 2-3 User Stories в формате Gherkin с позитивными и негативными сценариями

**Пример R.C.T.F. промпта:**

```
Role: Ты опытный Business Analyst, 
специализирующийся на BDD 
(Behavior Driven Development).
Context: Для проекта WeatherService 
(REST API) нужно формализовать требования 
для команды разработки и QA.
Task: Преобразуй User Story "Как пользователь 
API, я хочу подписаться на погоду в городе 
через POST /subscribe" в формат Gherkin. 
Создай:
- 1 позитивный сценарий (успешная подписка 
через API)
- 2 негативных сценария (город не найден, 
email уже подписан)
Format: Gherkin синтаксис (.feature файл) 
с Feature, Scenario, Given/When/Then.
```

## Задание 3: DoR и DoD v2.0

**Цель:** структурировать чек-листы по категориям

**Категории:**

- Requirements (требования)
- Technical (технические аспекты)
- Design (дизайн API/контракты)
- Testing (тестирование)
- Documentation (документация)

**Результат:** DoR v2.0 и DoD v2.0 (минимум 8 пунктов каждый)

**Пример R.C.T.F. промпта для DoR:**

```
Role: Ты опытный Product Owner 
с 5-летним опытом в Agile/Scrum.
Context: Мы разрабатываем WeatherService — 
REST API для уведомлений о погоде.
Наш текущий Definition of Ready v1.0 
слишком общий и не структурирован.
Task: Создай улучшенную версию 
Definition of Ready v2.0, 
структурированную по категориям:
- Requirements (требования к задаче)
- Technical (технические аспекты)
- Design (дизайн API/контракты)
- Testing (тестирование)
- Documentation (документация)
Format: Структурированный чек-лист 
в Markdown. Каждая категория должна 
содержать 3-5 конкретных пунктов.
```

## Задание 4: План тестирования

**Цель:** создать профессиональный тест-план

**Пирамида тестирования:**

- Unit тесты (функции)
- Integration тесты (связи между модулями)
- E2E тесты (весь флоу)

**Результат:** Структурированный тест-план (минимум 6 тест-кейсов)

**Пример R.C.T.F. промпта:**

```
Role: Ты Senior QA Engineer 
с опытом в тестировании REST API.
Context: Проект WeatherService (REST API) 
требует полного покрытия тестами.
Task: Создай тест-план, структурированный 
по пирамиде тестирования:
- 2 Unit теста (тестирование функций 
валидации города, кэширования)
- 2 Integration теста (интеграция с 
Weather API, работа с PostgreSQL)
- 2 E2E теста (полный флоу: 
POST /subscribe → GET /weather/{city})
Format: Markdown таблица с колонками: 
Test ID | Level | Preconditions | 
Steps | Expected Result
```

## Задание 5: Functional Delivery v2.0

**Цель:** улучшить Jira-тикеты с детализацией

**Добавить:**

- Детальные Acceptance Criteria
- Зависимости между тикетами
- Приоритеты (High/Medium/Low)
- Оценки времени
- Детальные тест-кейсы

**Результат:** 3-4 улучшенных Jira-тикета

**Пример R.C.T.F. промпта:**

```
Role: Ты Senior Delivery Manager 
с опытом в Agile и управлении бэклогом.
Context: У нас есть базовые Jira-тикеты 
для WeatherService v1.0 из Практики 1.
Нужно улучшить их до профессионального уровня.
Task: Улучши следующие тикеты:
- Добавь детальные Acceptance Criteria 
в формате Given/When/Then
- Укажи зависимости между тикетами
- Добавь приоритеты (High/Medium/Low)
- Добавь оценки времени (story points 
или часы)
- Детализируй тест-кейсы
Format: Markdown таблица или 
структурированный список тикетов.
Каждый тикет должен содержать: Title, 
Description, AC, Test Cases, 
Dependencies, Priority, Estimate.
```
