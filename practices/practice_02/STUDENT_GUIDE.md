## 📋 Модель R.C.T.F. — краткая памятка

### Структура идеального промпта

```
[R - Role] Действуй как [роль с опытом]

[C - Context] Контекст проекта:
- Что мы делаем
- Какие технологии используем
- Какие ограничения есть

[T - Task] Конкретная задача: [глагол действия]

[F - Format] Формат результата:
- Тип документа
- Структура
- Детали оформления
```

### Примеры ролей

- Senior DevOps Architect
- QA Automation Engineer
- Test Lead
- Product Manager
- Junior Developer (для критики)
- Security Engineer

### Примеры задач (глаголы)

- Сгенерируй
- Создай
- Модифицируй
- Напиши
- Детализируй
- Проанализируй

## 📝 Пошаговая инструкция

### Подготовка

**Что делать:**

```
1. Соберитесь в ту же группу (3-4 человека)
2. Откройте результаты Практики 1:
   - Ваши User Stories
   - Mermaid диаграмму
   - План тестирования
3. Откройте `task.py` в редакторе — это ваша рабочая тетрадь
4. **Что открыть в `task.py`:**
   - `STUDENT_INFO` — заполните информацию о группе
   - `PROMPT_LOGS` — журнал R.C.T.F. промптов (класс `RCTF_Log`)
   - `MERMAID_V2` — для улучшенной архитектурной схемы
   - `GHERKIN_SCENARIOS` — для формализованных требований в Gherkin
   - `DOR_V2` — для улучшенного Definition of Ready
   - `DOD_V2` — для улучшенного Definition of Done
   - `TEST_PLAN_V2` — для структурированного плана тестирования
   - `FUNCTIONAL_DELIVERY_V2` — для улучшенных Jira-тикетов
   - `HOMEWORK_EVENT_STORMING_V2` — опционально: улучшенный Event Storming
   - `HOMEWORK_ROADMAP_V2` — опционально: улучшенный Roadmap
   - `REFLECTION` — для рефлексии
5. Откройте LLM-чат (новая сессия!)
6. Откройте доску для визуализации
```

**Создайте на доске новые разделы:**

- "Архитектура v2.0"
- "Бэклог v2.0"
- "Тестирование v2.0"

**Для каждого артефакта будет:**

- Старая версия (v1.0) слева
- Новая версия (v2.0) справа
- Промпт, который привел к улучшению

```
[C] У нас есть User Story:
"[вставьте вашу User Story]"

Технический контекст:
- Система: REST API на Python (FastAPI)
- API погоды: OpenWeatherMap
- База данных: PostgreSQL (хранит: email, city, notification_time)
- Кэширование: Redis (кэш данных о погоде с TTL 10 минут)

Пользовательский флоу через API:
1. Клиент отправляет POST /subscribe с {city, email}
2. API проверяет существование города через OpenWeatherMap
3. API сохраняет подписку в PostgreSQL
4. API возвращает подтверждение с данными о погоде

[F] Используй строгий Gherkin-синтаксис (Given/When/Then).

Scenario 1: [Название позитивного сценария]
  Given [предусловие]
  When [действие]
  Then [ожидаемый результат]
  And [дополнительная проверка]

Scenario 2: [Название негативного сценария]
  ...

Требования:
- Минимум 2 позитивных сценария
- Минимум 2 негативных сценария (несуществующий город, дубликат подписки)
- 1 граничный случай (город с пробелами/спецсимволами)
- Итого: минимум 5 сценариев
```

**Что делать:**

```
1. Скопируйте шаблон
2. Вставьте вашу User Story
3. Уточните технические детали (если отличаются)
4. Отправьте промпт
5. Получите AC в формате Gherkin
```

#### Шаг 3: Проверка Gherkin

**Чек-лист Gherkin:**

```
☐ Каждый сценарий начинается с "Scenario:"
☐ Есть Given (предусловия)
☐ Есть When (действие)
☐ Есть Then (ожидаемый результат)
☐ And/But используется для дополнительных шагов
☐ Формулировки четкие и конкретные
☐ Нет технических деталей реализации (только поведение)
```

**Пример правильного Gherkin:**

```
Scenario: Успешная подписка на уведомления для существующего города через API
  Given клиент имеет валидный API endpoint
  And email "user@test.com" не подписан на уведомления
  When клиент отправляет POST /subscribe с {"city": "Moscow", "email": "user@test.com"}
  Then API проверяет город через OpenWeatherMap API
  And API сохраняет подписку в базе данных PostgreSQL
  And API возвращает 200 OK с {"message": "Subscribed to Moscow", "weather": {...}}
  And данные о погоде кэшируются в Redis на 10 минут
```

**Если Gherkin неправильный:**

```
Уточняющий промпт:
"Перепиши сценарии, используя строгий Gherkin-синтаксис.
Каждый шаг должен начинаться с Given/When/Then/And/But.
Убери технические детали реализации, оставь только видимое поведение."
```

**Куда сохранить:**

- AC в Gherkin → `task.py` → `GHERKIN_SCENARIOS`
- Финальный промпт → `task.py` → `RCTF_Log` (добавьте в список PROMPT_LOGS)

## ⚠️ Частые проблемы и решения

### Проблема: LLM игнорирует формат

**Решение:**

```
Сделайте секцию [F] более детальной:

[F] Markdown-таблица. ВАЖНО: строго соблюдай формат.
Структура таблицы:
| ID | Тип | Описание | Шаги | Результат |
|----|----|----------|------|-----------|
| TC-001 | Unit | ... | ... | ... |

Требования:
- Первая строка: заголовки колонок
- Вторая строка: разделители (|----|----|...)
- Далее: строки с данными
- Минимум 10 строк данных
```

### Проблема: Роль не влияет на ответ

**Решение:**

```
Усильте роль, добавив детали:

❌ Плохо: "Действуй как архитектор"
✅ Хорошо: "Действуй как Senior DevOps Architect с 10-летним опытом 
             в проектировании высоконагруженных микросервисных систем,
             специализирующийся на Python и message-driven архитектурах"
```

### Проблема: Контекст слишком большой

**Решение:**

```
Разбейте задачу на мультипромптинг:

Промпт 1: Общая архитектура
Промпт 2: Детализация компонента 1
Промпт 3: Детализация компонента 2
```

### Проблема: Gherkin синтаксис неправильный

**Решение:**

```
Добавьте примеры в формат:

[F] Gherkin-синтаксис. Пример правильного сценария:

Scenario: Успешная подписка
  Given пользователь авторизован
  And пользователь не имеет активных подписок
  When пользователь отправляет команду /subscribe
  And вводит город "Москва"
  Then система создает подписку
  And пользователь получает подтверждение

Создай минимум 5 таких сценариев для нашей User Story.
```
