# Практика 01: От идеи до плана разработки с AI

**Связь с лекцией:** Engineer Journey Map (EJM), этап Planning & Analysis

## 🎯 Цель практики

Пройти путь **от идеи продукта до набора артефактов**, которые можно отдать команде разработки на оценку и реализацию: бизнес-логика/роадмап → декомпозиция → архитектура → качество.

## 📋 Проект: WeatherService

WeatherService — REST API сервис уведомлений о погоде:
- подписка на города и управление подписками;
- получение текущей погоды/прогноза из внешнего Weather API;
- клиенты: веб/мобильные приложения (опционально: Telegram Bot).

## 📦 Итоговые артефакты (что сдаём)

Артефакты оформляются в `practices/practice_01/task.py` и попадают в итоговый отчёт `artifacts/report_p1.md`.

- **Event storming (light)**: actors/commands/events
- **CJM**: основной путь пользователя
- **Roadmap**: v1.0/v1.1/v2.0 (слайс по ценности)
- **Epics + User Stories**
- **DoR (Definition of Ready)**
- **ADR + Mermaid (HLD)** + ссылка/путь на изображение (если сохраняли)
- **DoD (Definition of Done) + тест-план**
- **Functional Delivery**: нарезка задач в стиле Jira-тикетов (с AC и тест-кейсами)
- **PROMPT_LOGS + REFLECTION**

## 🕐 Структура практики

### 1) Вступление

- **Цель**: задать рамку и ожидания.
- **Что говорит преподаватель**:
  - «Сегодня мы соберём план разработки: от идеи до артефактов, пригодных для оценки командой».
  - «AI помогает быстро "упаковать" мысли: CJM/роадмап → Epics/User Stories → ADR/HLD → QA/Delivery».
- **Инструкции**:
  - открыть `task.py` — рабочую тетрадь (в файле есть секции для каждого артефакта: `EVENT_STORMING`, `CJM`, `ROADMAP`, `EPICS`, `USER_STORIES`, `DEFINITION_OF_READY`, `ARCHITECTURE_COMPONENTS`, `ADR`, `MERMAID_CODE`, `DEFINITION_OF_DONE`, `TEST_PLAN`, `FUNCTIONAL_DELIVERY`, `PROMPT_LOGS`, `REFLECTION`);
  - подготовить доступ к LLM и доску (Miro/Excalidraw/Draw.io).

### 2) Задание 1 — CJM и UX через event storming

- **Цель**: формализовать идею в понятный путь пользователя и разложить на версии (roadmap).
- **Куда писать результат в `task.py`**:
  - `EVENT_STORMING`, `CJM`, `ROADMAP`, `EPICS`, `USER_STORIES`, `DEFINITION_OF_READY`
- **Ожидаемый результат**:
  - 5–7 events, 3–5 commands, 3–5 actors
  - CJM на 6–8 шагов
  - roadmap v1.0/v1.1/v2.0
  - 4–6 epics, 5–7 user stories
  - DoR на 4–6 пунктов

### 3) Задание 2 — ADR + HLD

- **Цель**: перейти от бизнес-артефактов к техническому решению и зафиксировать его формально.
- **Куда писать результат в `task.py`**:
  - `ARCHITECTURE_COMPONENTS`, `ADR`, `MERMAID_CODE`, `MERMAID_IMAGE_URL`
- **Ожидаемый результат**:
  - краткий ADR (контекст/решение/альтернативы/последствия)
  - валидная Mermaid диаграмма с компонентами и связями

### 4) Задание 3 — QA + Functional Delivery

- **Цель**: сформировать профессиональную «нарезку на delivery» и качество.
- **Куда писать результат в `task.py`**:
  - `DEFINITION_OF_DONE`, `TEST_PLAN`, `FUNCTIONAL_DELIVERY`
- **Ожидаемый результат**:
  - DoD на 4–6 пунктов
  - тест-план: минимум 8 тест-кейсов (позитивные/негативные)
  - 6–10 Jira-тикетов с AC и тестами

### 5) Демонстрация «До/После»

- **Что происходит**: 1–2 команды показывают улучшенные артефакты и финальные промпты.

### 6) Завершение и отчёт

- **Что сделать**:
  - заполнить `PROMPT_LOGS` и `REFLECTION`;
  - запустить `python3 task.py`;
  - убедиться, что сформировался `artifacts/report_p1.md`.
