# Feature Specification: WeatherService REST API

**Feature Branch**: `main`  
**Created**: 2026-04-26  
**Status**: Implemented  
**Input**: Учебный проект ИТМО «Практика 03» — REST API для уведомлений о погоде

## User Scenarios & Testing

### User Story 1 - Получение текущей погоды (Priority: P1)

Пользователь хочет узнать текущую погоду в любом городе мира через простой API запрос.

**Why this priority**: Базовая функциональность, которая демонстрирует интеграцию с внешним API и является основой для всех остальных функций.

**Independent Test**: Можно полностью протестировать отправкой GET запроса на `/weather/{city}` и проверкой структуры ответа с температурой, влажностью и описанием погоды.

**Acceptance Scenarios**:

1. **Given** пользователь знает название города, **When** отправляет GET `/weather/Moscow`, **Then** получает JSON с температурой, влажностью и описанием погоды (200 OK)
2. **Given** пользователь вводит несуществующий город, **When** отправляет GET `/weather/NonExistentCity123`, **Then** получает ошибку "City not found" (404)
3. **Given** внешний API недоступен, **When** отправляет запрос, **Then** получает ошибку "Weather service temporarily unavailable" (503)

---

### User Story 2 - Создание подписки на уведомления (Priority: P2)

Пользователь хочет подписаться на ежедневные уведомления о погоде для выбранного города, указав свой email.

**Why this priority**: Ключевая бизнес-функция, которая отличает сервис от простого прокси к OpenWeatherMap. Требует валидации данных и работы с базой данных.

**Independent Test**: Можно протестировать отправкой POST `/subscribe` с email и городом, проверкой создания записи в БД и получением subscription_id в ответе.

**Acceptance Scenarios**:

1. **Given** пользователь с валидным email, **When** отправляет POST `/subscribe` с `{"email": "user@example.com", "city": "London"}`, **Then** создаётся подписка и возвращается subscription_id с текущей погодой (201 Created)
2. **Given** пользователь уже подписан на город, **When** пытается подписаться повторно на тот же город, **Then** получает ошибку "Already subscribed" (409 Conflict)
3. **Given** пользователь вводит невалидный email, **When** отправляет запрос с `{"email": "invalid", "city": "Paris"}`, **Then** получает ошибку валидации (400 Bad Request)
4. **Given** пользователь вводит несуществующий город, **When** отправляет запрос, **Then** получает ошибку "City not found" (404)

---

### User Story 3 - Просмотр всех подписок (Priority: P3)

Пользователь (или администратор) хочет видеть список всех активных подписок в системе.

**Why this priority**: Вспомогательная функция для мониторинга и отладки. Не критична для MVP, но полезна для управления.

**Independent Test**: Можно протестировать отправкой GET `/subscriptions` и проверкой структуры ответа со списком подписок.

**Acceptance Scenarios**:

1. **Given** в системе есть подписки, **When** отправляет GET `/subscriptions`, **Then** получает список всех подписок с email, городом и ID (200 OK)
2. **Given** в системе нет подписок, **When** отправляет GET `/subscriptions`, **Then** получает пустой список (200 OK)

---

### User Story 4 - Удаление подписки (Priority: P3)

Пользователь хочет отписаться от уведомлений о погоде для конкретного города.

**Why this priority**: Необходимая функция для управления подписками, но не блокирует основной функционал.

**Independent Test**: Можно протестировать созданием подписки, затем отправкой DELETE `/subscribe/{id}` и проверкой, что подписка удалена из БД.

**Acceptance Scenarios**:

1. **Given** существует подписка с ID=5, **When** отправляет DELETE `/subscribe/5`, **Then** подписка удаляется (204 No Content)
2. **Given** подписка с указанным ID не существует, **When** отправляет DELETE `/subscribe/999`, **Then** получает ошибку "Subscription not found" (404)

---

### Edge Cases

- Что происходит, когда OpenWeatherMap API возвращает таймаут (>5 секунд)? → 503 Service Unavailable
- Что происходит, когда OpenWeatherMap API возвращает неожиданный статус код (не 200/404)? → 503 Service Unavailable
- Что происходит, когда пользователь пытается подписаться на город с названием из 1 символа? → 400 Bad Request (минимум 2 символа)
- Что происходит, когда пользователь пытается подписаться на город с названием >100 символов? → 400 Bad Request
- Что происходит при попытке удалить подписку с невалидным ID (не число)? → 422 Unprocessable Entity (FastAPI автоматически)
- Что происходит при concurrent запросах на создание одинаковой подписки? → UniqueConstraint в БД предотвращает дубликаты, один запрос получит 409

## Requirements

### Functional Requirements

- **FR-001**: Система ДОЛЖНА предоставлять эндпоинт GET `/weather/{city}` для получения текущей погоды
- **FR-002**: Система ДОЛЖНА возвращать температуру в градусах Цельсия, влажность в процентах и текстовое описание погоды
- **FR-003**: Система ДОЛЖНА предоставлять эндпоинт POST `/subscribe` для создания подписки на уведомления
- **FR-004**: Система ДОЛЖНА валидировать email адрес при создании подписки
- **FR-005**: Система ДОЛЖНА валидировать название города (2-100 символов)
- **FR-006**: Система ДОЛЖНА проверять существование города через OpenWeatherMap API перед созданием подписки
- **FR-007**: Система ДОЛЖНА предотвращать создание дубликатов подписок (один email + один город)
- **FR-008**: Система ДОЛЖНА автоматически создавать пользователя при первой подписке (get_or_create pattern)
- **FR-009**: Система ДОЛЖНА предоставлять эндпоинт DELETE `/subscribe/{id}` для удаления подписки
- **FR-010**: Система ДОЛЖНА предоставлять эндпоинт GET `/subscriptions` для получения списка всех подписок
- **FR-011**: Система ДОЛЖНА возвращать текущую погоду в ответе на успешное создание подписки
- **FR-012**: Система ДОЛЖНА логировать каждый входящий запрос и его исход
- **FR-013**: Система ДОЛЖНА использовать структурированные JSON ответы для всех эндпоинтов
- **FR-014**: Система ДОЛЖНА возвращать понятные сообщения об ошибках с соответствующими HTTP статус кодами

### Key Entities

- **User**: Представляет пользователя системы
  - Атрибуты: id (уникальный идентификатор), email (уникальный email адрес), created_at (дата создания)
  - Создаётся автоматически при первой подписке
  - Связь: один пользователь может иметь много подписок

- **Subscription**: Представляет подписку пользователя на уведомления о погоде
  - Атрибуты: id (уникальный идентификатор), user_id (ссылка на пользователя), city (название города), notification_time (время уведомления, по умолчанию 09:00), is_active (флаг активности), created_at (дата создания)
  - Ограничение: уникальная пара (user_id, city) — один пользователь не может подписаться на один город дважды
  - Связь: принадлежит одному пользователю, при удалении пользователя подписки удаляются каскадно

- **WeatherData**: Представляет данные о погоде (не хранится в БД, получается из внешнего API)
  - Атрибуты: city (название города), temperature (температура в °C), humidity (влажность в %), description (описание погоды)
  - Источник: OpenWeatherMap API

## Success Criteria

### Measurable Outcomes

- **SC-001**: Пользователи могут получить данные о погоде для любого города за время < 6 секунд (5 сек таймаут API + обработка)
- **SC-002**: Система корректно обрабатывает 100% невалидных входных данных с понятными сообщениями об ошибках
- **SC-003**: Система предотвращает 100% попыток создания дубликатов подписок
- **SC-004**: Все API эндпоинты возвращают структурированные JSON ответы согласно OpenAPI схеме
- **SC-005**: Система логирует 100% запросов для возможности отладки и мониторинга
- **SC-006**: Тестовое покрытие составляет 100% критических путей (happy path + основные ошибки)
- **SC-007**: Система корректно обрабатывает недоступность внешнего API (таймауты, ошибки сети)
- **SC-008**: База данных поддерживает целостность данных через constraints (UNIQUE, FOREIGN KEY, CASCADE)

## Assumptions

- Пользователи имеют стабильное интернет-соединение для доступа к API
- OpenWeatherMap API доступен и имеет актуальные данные о погоде
- Названия городов вводятся на английском языке (требование OpenWeatherMap API)
- Фактическая отправка email уведомлений не реализована в текущей версии (только хранение подписок)
- Аутентификация пользователей не требуется (публичный API)
- Пагинация и фильтрация для GET `/subscriptions` не требуется в MVP
- Система работает в одном часовом поясе (notification_time без timezone)
- PostgreSQL база данных доступна и настроена через Docker
- API ключ OpenWeatherMap предоставлен и активирован
- Система развёртывается в окружении с Python 3.10+
- Миграции базы данных выполняются через `create_all()` при старте (Alembic не используется)
- Concurrent запросы обрабатываются корректно благодаря async/await и database constraints
- Система предназначена для учебных целей, production-ready features (rate limiting, caching, monitoring) не требуются

## Non-Functional Requirements

### Performance
- Таймаут для запросов к OpenWeatherMap API: 5 секунд
- Асинхронная обработка всех I/O операций (БД, HTTP)

### Security
- API ключи хранятся в переменных окружения, не в коде
- Валидация всех входных данных через Pydantic
- SQL injection предотвращается через ORM (SQLAlchemy)

### Reliability
- Graceful handling внешних ошибок (таймауты, недоступность API)
- Database constraints обеспечивают целостность данных
- Каскадное удаление подписок при удалении пользователя

### Maintainability
- Чёткое разделение ответственности (роутеры, репозитории, модели)
- Type hints для всех функций
- Структурированное логирование
- Автоматическая документация через OpenAPI/Swagger

### Testability
- Все внешние зависимости мокируются в тестах
- Независимые unit тесты для каждого эндпоинта
- Покрытие happy path и error cases
