# Product Context

## Зачем существует этот продукт

WeatherService решает задачу получения актуальной погоды и управления подписками на уведомления. Пользователь указывает email и город — сервис проверяет город через OpenWeatherMap, сохраняет подписку и возвращает текущую погоду.

## Пользовательские сценарии

### Сценарий 1: Подписка на погоду
1. Пользователь отправляет `POST /subscribe` с `{"email": "user@example.com", "city": "Moscow"}`
2. Сервис проверяет, что город существует (через OpenWeatherMap)
3. Сервис проверяет, что подписка ещё не существует
4. Создаёт запись в БД, возвращает `201` с данными о погоде и `subscription_id`

### Сценарий 2: Получение погоды
1. Пользователь отправляет `GET /weather/London`
2. Сервис запрашивает OpenWeatherMap API
3. Возвращает `{"city": "London", "temp": 12.5, "humidity": 80, "description": "overcast clouds"}`

### Сценарий 3: Управление подписками
- `GET /subscriptions` — просмотр всех активных подписок
- `DELETE /subscribe/{id}` — удаление конкретной подписки

## Бизнес-правила

| Правило | Реализация |
|---------|-----------|
| Один пользователь не может подписаться на один город дважды | `UNIQUE(user_id, city)` в БД + проверка в роутере → 409 |
| Город должен существовать в OpenWeatherMap | Вызов `_fetch_weather` перед созданием подписки → 404 |
| Email должен быть валидным | `EmailStr` в Pydantic → 400 |
| Название города: 2–100 символов | `Field(min_length=2, max_length=100)` → 400 |

## UX API

- Все ошибки возвращают `{"detail": "..."}` (модель `ErrorResponse`)
- Успешное создание подписки возвращает погоду сразу — не нужен отдельный запрос
- `GET /subscriptions` всегда возвращает 200 (пустой список — не ошибка)
- Пользователь создаётся автоматически при первой подписке (нет отдельного эндпоинта регистрации)
